Overview

This document covers the full automation architecture behind the SEO team's monthly task generation system. The goal was to replace a manual, error-prone monthly process with a scheduled system that required zero human input. Every Jira work item for every client needed to exist on the first of the month, already assigned, so the team could start work immediately.

The system uses 22 total automation rules: one scheduled trigger, six task creation automations, and fifteen user assignment automations. All rules live in a dedicated Jira Software project. The Premium tier's high automation limit made it practical to keep rules separated by function rather than consolidating them into dense, hard-to-read logic chains.

Environment

The SEO team had two dedicated Jira projects. The Jira Software project handled all monthly deliverable work, the kanban board the team operated from day to day. A separate Jira Service Management project handled inbound requests from other teams. This architecture document covers the Software project only.

Issue type setup

Jira's default Epic issue type was replaced with three custom epic-level types: PackageA, PackageB, and Custom. These sit at the epic tier of the work item hierarchy and function as the parent for all monthly task work items underneath them.

Using custom types instead of the default Epic gave the team an immediate visual signal about the client's service tier when looking at the board or backlog. It also gave the automation rules a clean, unambiguous trigger condition, each automation checks for a specific issue type rather than reading a field value.

All child task work items (Blog Post, Keyword Research, On-Page Optimization, etc.) sit at the story level of the hierarchy and are linked to their parent epic via the parent field. This creates the full branching structure: one epic per client per month, with all that client's tasks nested underneath it.

Note on hierarchy: Jira's work item hierarchy (Epic → Story → Sub-task) is not strictly enforced by the platform. The parent field is what creates the relationship. This means you can link any issue type to any other as a parent, which is how custom epic-level types like PackageA, PackageB, and Custom can parent standard story-level task types.

Scheduled trigger

One automation runs on a scheduled trigger on the first of every month. It creates one epic-level work item per client. Two fields are set at creation time:

  • Issue type, set to PackageA, PackageB, or Custom depending on the client's service tier.
  • Client ID, a custom dropdown field that holds the unique identifier for that client. This is the key that all downstream automations read to determine quantities and assignments.

The creation of these epics is what triggers every subsequent automation. PackageA and PackageB task creation automations listen for a new issue of their respective type. Custom task creation automations do the same. Nothing fires until an epic exists.

PackageA automation

Trigger: when a PackageA issue is created at the epic tier.

Action: creates the full fixed task set for that client. Every PackageA client gets the same tasks every month, no Client ID check required. The automation generates approximately 10 work items covering a curated blend of SEO task types considered sufficient for the entry-level tier.

All child tasks are linked to the parent PackageA epic via the parent field, creating the nested structure visible on the board and in the backlog.

Why no Client ID check? PackageA and PackageB are fixed packages. Every client on the tier gets the same deliverables. Client ID only matters for assignment (which team member gets the task) and for Custom clients (how many of each task type to generate). Task creation for standard tiers is entirely uniform.

PackageB automation

Trigger: when a PackageB issue is created at the epic tier.

Action: same pattern as PackageA. One automation generates the full fixed task set, approximately 20 work items covering most or all available task types, with some types appearing multiple times. No Client ID check. Every PackageB client gets an identical task set each month.

PackageB is the full-service tier. The task set was designed to cover every deliverable type a client could expect when they bought the complete offering.

Custom automations

Custom clients have bespoke monthly packages, any combination of the available task types, any quantity, built per client. This is where the automation complexity lives.

Trigger for all four: when a Custom issue is created at the epic tier. All four automations fire simultaneously in parallel.

Why four automations?

One automation hit Jira's object limit, the maximum number of actions a single rule can perform. With 45+ clients, each potentially needing multiple instances of the same task type, a single automation would have run out of capacity. Splitting the logic across four automations kept each one well within the limit and left room to add new task types later without rebuilding anything.

How the if/then logic works

Each automation contains a series of if/then statements. The condition checks two things: the Client ID field on the epic, and how many of a given task type that client needs. If a client needs three Blog Posts, the automation has three separate if statements, one for each, and only the Client IDs requiring that many posts will match each condition.

Tasks with multiples are titled sequentially: Blog Post #1, Blog Post #2, Blog Post #3. This keeps the board readable and makes it obvious which tasks belong to the same deliverable group.

Assignment automations

Once tasks are created, a separate set of automations handles assignment. There are approximately 15 assignment rules, each triggered when a specific issue type is created.

Assignment logic is package-agnostic. It does not check whether a task came from PackageA, PackageB, or Custom, it only cares about what the task is (issue type, and sometimes ticket title for variations of the same type) and who the client is (Client ID).

Most task types are split across 2-3 team members depending on the client. Each assignment automation uses an if/then/else chain that covers every Client ID, assigning the correct person for that client. The if/then/else structure means every client has exactly one matching condition and gets one assignee, no ambiguity, no fallthrough.

Keeping assignment separate from task creation was a deliberate decision. It made the rules easier to read, easier to update when team members changed, and easier to troubleshoot if something assigned incorrectly. You could open an assignment automation, find the Client ID, and immediately see who was assigned and why.

Client configuration template

Managing Custom client configurations through meetings and informal requests created problems. The manager struggled to communicate changes clearly, which led to tasks being generated incorrectly and back-and-forth that slowed everything down.

The fix was a simple Excel template. The structure:

  • Column A: Client ID
  • Columns B onward: One column per task type (Blog Post, Keyword Research, On-Page Optimization, etc.)
  • Cell values: The number of that task type to generate for that client each month

When a client's package changed, the manager filled in the row for that Client ID and handed the template to me. I updated the if/then logic in the relevant Custom automation to match. The template was also structured to mirror the four-automation split, so I could immediately see which automation needed updating without cross-referencing anything.

It was a simple tool, but it eliminated the communication problem entirely. Changes went from a source of frustration to a five-minute update.

Download the template: A blank version of this configuration template is available as a downloadable artifact from the P-002 overview page. The structure matches what was used in production, with generic column headers in place of internal terminology.

Ongoing maintenance

After go-live, three categories of maintenance came up regularly:

  • Client roster changes: Adding a new Custom client meant adding their Client ID to the relevant if/then blocks across all four Custom automations and the relevant assignment automations. Removing a client meant removing those blocks. Standard tier additions required only adding to the scheduled trigger.
  • Team changes: When someone joined or left the SEO team, the assignment automations for the issue types they handled needed updating. Because assignment logic lived in separate, clearly labeled automations, changes were fast to find and make.
  • Platform outages: Occasionally Jira's global automation infrastructure had issues that caused rules not to fire. This was outside my control. When it happened, I ran the affected automations manually for impacted clients and documented the outage.

Outside of these three scenarios, the system ran without intervention from go-live through my departure.
