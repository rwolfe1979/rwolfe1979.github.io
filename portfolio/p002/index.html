SEO Team Monthly Automation

The SEO team was spending days every month manually creating the same Jira tickets for every client. I shadowed one full cycle, documented every step, and built a scheduled automation that generates the entire monthly work set on the first of each month. They stopped thinking about it.

Details in this project have been generalized to protect confidential information. The system described is based on work I designed and built in a production environment.

The Problem

Every month, the SEO team manager manually created every Jira ticket for every client. With around 10 clients at the start, that process took a couple of days — squeezed between their actual work. The task set was predictable: same types, same assignees, same due date logic month after month. The Custom tier added complexity because each client had a different combination of tasks and quantities that changed over time.

As the client roster grew, doing this manually would have taken a week or more per month. It was the kind of problem that only gets worse.

Automation Flow

On the first of each month, a scheduled trigger creates one epic-level work item per client. The package type on that epic determines which automations fire. PackageA and PackageB each run a single automation producing a fixed task set. Custom clients trigger four automations running in parallel, each handling a subset of the possible task types with if/then logic that checks the Client ID to determine quantities.

What I Built
01
Custom issue type hierarchy
Created three custom epic-level issue types — PackageA, PackageB, and Custom — so team members could see a client's service tier at a glance without opening the issue. Standard Jira "Epic" was replaced with meaningful types that matched how the team actually thought about their work.
02
Scheduled epic creation automation
A single automation runs on the first of every month and creates one epic-level work item per client. Package type and Client ID are set directly on the epic, which is what every downstream automation reads to decide what to do next.
03
PackageA and PackageB child task automations
One automation per tier. When a PackageA or PackageB epic is created, the corresponding automation fires and generates the full fixed task set. No Client ID logic needed — every client on a standard tier gets the same tasks every month.
04
Four Custom child task automations
Custom clients needed variable quantities of each task type. One automation hit Jira's object limit, so I split the logic across four automations running in parallel. Each handles a subset of task types, using if/then statements keyed on Client ID to determine how many tickets to generate. Tasks with multiples are titled sequentially (Blog Post #1, Blog Post #2, etc.).
05
User assignment automations
About 15 separate automations handle assignment. Each checks issue type (and sometimes the ticket title for variations) and Client ID, then assigns the right team member. Tasks were typically split across 2-3 people per issue type, so each automation used if/then/else chains to cover every client. Assignment logic is package-agnostic — it only cares about what the task is and who the client is.
06
Client configuration template
The Custom tier required a process for managing changes. When a client's needs shifted, the manager needed a clean way to communicate updates without ambiguity. I built a simple Excel template — one row per Client ID, one column per task type, number in the cell equals tickets to generate. Updates went from frustrating back-and-forth to a five-minute form fill.

The Outcome

What had been taking the manager a couple of days each month for 10 clients — and would have scaled to a week or more at 45+ clients — became fully automated. Every month on the first, every client's full task set appeared in Jira already assigned. The team started work the same day with no setup required.

The system ran without issues from go-live through my departure. Ongoing maintenance was limited to updating Custom client configurations when service packages changed and adjusting assignment automations when team members joined or left.
