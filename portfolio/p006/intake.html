1. Purpose

This document defines the process for requesting a new Jira project, the standards every project must meet, and the criteria used to evaluate whether a project request is appropriate. It exists so that new projects are consistent, maintainable, and actually useful to the teams that use them.

2. Scope

Applies to all Jira Software and Jira Service Management project requests across all departments. Any team requesting a new Jira project follows this process, regardless of department or seniority of the requestor.

3. When to Request a New Project

Not every workflow needs its own Jira project. Before submitting a request, the requestor should confirm that the work meets at least one of the following criteria:

  • The work involves a distinct team, department, or cross-functional group with its own workflows and ownership
  • The work has a defined lifecycle — requests come in, get assigned, get resolved, and close
  • There is a need to track status, priority, or accountability that cannot be handled in an existing project
  • The team will actively manage and maintain the project on an ongoing basis

If the request is better served by a board in an existing project, a new issue type, or a Confluence page, the administrator will advise the requestor accordingly before any configuration work begins.

4. Intake Process

4.1 Submit a Request

All new project requests are submitted through the Jira Request Portal — a dedicated JSM service desk maintained independently by the Jira administrator, separate from the general IT service desk.

4.2 Review and Approval

The Jira administrator reviews each request within 3 business days. The review checks that:

  • The request includes complete information
  • The use case is appropriate for Jira and can't be met by an existing project
  • The proposed naming and structure follows platform standards
  • The permission level requested aligns to an existing tier and can be handled through the standard Permission Group structure
  • The request doesn't create a permission or governance conflict with existing projects

If approved, the administrator schedules configuration and notifies the project lead. If the request needs clarification or the use case is better served another way, the administrator contacts the requestor directly before any work begins.

4.3 Configuration and Handoff

Once approved, the administrator configures the project according to the standards in Section 5 and the specific requirements captured in the request. This includes issue types and request type categories, screen and field configuration, workflow setup, and Permission Group assignment.

Before go-live, the project lead is walked through the configuration, confirms it matches their requirements, and signs off. The administrator documents the project in the internal project registry.

6. Standard Configuration Applied to All Projects

Every new project is configured with the following baseline settings. Deviations from these standards require documented justification and administrator sign-off.

6.1 Issue Types and Request Type Categories

Issue types are configured in two steps for both project types, starting with broad categories before defining the specific types within them.

Jira Service Management

Step 1 — Define request categories. Incoming work is first grouped into broad categories that reflect the types of requests the service desk handles (for example: Access Requests, Hardware, Software, Onboarding, Reporting). Categories make the customer portal easier to navigate and allow issue types to be reused across similar request types.

Step 2 — Assign or build issue types per category. Each category maps to one or more issue types. A single issue type can back multiple request types within the same category. Standard issue types (Service Request, Incident, Change) are used where they fit. Custom issue types are built when the standard set doesn't match the team's workflow.

Jira Software

Step 1 — Define work categories. Work is grouped by type: planned work (features, stories), unplanned work (bugs, incidents), operational tasks, and sub-tasks. Categories reflect how the team actually thinks about their workload.

Step 2 — Assign issue types per category. Standard issue types (Story, Task, Bug, Sub-task) are used where they fit. Custom issue types are added when the team's workflow requires a type that doesn't map cleanly to the standard set.

Both project types follow the same principle: categories come first, types come second. This prevents issue type sprawl and makes it easier to build screens and filters that work across similar projects.

6.2 Screens and Custom Fields

Screen configuration follows issue type setup. The goal is to reuse existing screens and custom fields wherever possible. New screens and fields are only created when the existing set genuinely can't meet the requirement.

  • Existing custom fields are audited before any new field is created. If a field already exists and collects the right data, it is reused even if it was originally built for a different project.
  • Screen reuse is attempted first. If an existing screen covers the fields needed for a new issue type, it is applied rather than duplicated.
  • When a custom screen is required, it is built to serve the specific issue type and configured for the correct contexts (create, edit, view) to avoid fields appearing where they're not relevant.
  • All custom fields are named clearly and documented with their purpose, field type, and which projects use them. This prevents duplicate fields being created under different names for the same purpose.

6.3 Workflows

Projects are configured with a workflow appropriate to the team's process. Standard workflows are applied where they fit. Custom workflows are built when the team requires statuses, transitions, or conditions that the standard set can't support.

Custom workflows are documented before deployment. The documentation covers each status, transition logic, and any conditions or validators applied. This ensures the workflow can be maintained or modified without relying on institutional knowledge.

6.4 Permission Scheme

All projects use a single universal permission scheme. Permissions are not configured per project. Instead, access is controlled through Permission Groups that map to standardized project roles. This approach keeps the permission model consistent across the environment and eliminates discrepancies between projects.

The universal scheme defines the following project role tiers:

Role TierWhat They Can Do
View OnlyRead access to the project — can see issues and comments but cannot create or edit
View and CommentCan read and add comments to existing issues, but cannot create new issues
View, Comment, and CreateCan read, comment, and create new issues — standard for stakeholders who submit requests
Member / AgentFull working access — can create, edit, transition, and resolve issues. Standard for team members and JSM agents.
LeadMember-level access plus the ability to manage components, versions, and basic project configuration. Cannot modify permission schemes or global settings.
Manager / AdminFull project administration rights within the project scope. Cannot modify global schemes or platform-level settings — those remain with the platform admin.

Permission Groups are assigned to these roles at the platform admin level. When a user is onboarded, they are added to the Permission Group that matches their position. This ensures they automatically have the correct access across all relevant projects without requiring individual project-level configuration.

Project leads rarely need to touch permissions directly. The administrator manages Permission Group assignments — a broader responsibility than what is typically expected of a project-level admin role.

6.5 Notification Scheme

All projects use a single universal notification scheme. Notification configuration is not done at the project level. This keeps notification behavior consistent across the environment and prevents the discrepancies that arise when individual teams customize their own schemes.

Teams can request adjustments to notification behavior through the standard change request process. Changes are evaluated against the universal scheme to determine whether a project-level override is warranted or whether the universal scheme should be updated for all projects.

7. Project Lead Responsibilities

The project lead named at intake is accountable for the following on an ongoing basis:

  • Keeping the project active — requesting archival or closure when the project is no longer in use
  • Flagging access changes to the administrator when team members change roles or leave the team
  • Escalating configuration change requests to the Jira administrator through the Jira Request Portal
  • Responding to quarterly access audits within the required timeframe

Project leads are not expected to manage permissions directly. Permission Group assignments are handled by the administrator. If a lead adds an individual user directly to a project role outside of the Permission Group structure, the administrator will review and standardize the assignment at the next audit.

8. Review and Maintenance

The Jira administrator conducts a quarterly review of all active projects. Projects with no activity in 90 days are flagged for review. Project leads are notified and given 10 business days to confirm the project should remain active or to request archival.
