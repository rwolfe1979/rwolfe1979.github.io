Permission Scheme Design and User Access Management

Every project in the environment started with its own auto-generated permission scheme. With 55+ projects across 20 departments, that meant 20+ schemes to manage, audit, and update individually. I replaced them all with a single universal scheme built around named role tiers and group-based access assignment. Permission changes that used to require touching dozens of schemes now take one edit.

Details in this project have been generalized to protect confidential information. The system described is based on work I designed and built in a production environment.

The Problem

When Jira creates a new project, it generates a permission scheme automatically. By default, that scheme belongs only to that project. For a single project, that's fine. At scale, it creates a management problem.

With 20+ auto-generated schemes across the environment, any change to how permissions worked had to be applied to each one manually. If Atlassian introduced a new permission type in a platform update, the only way to ensure every project had it configured correctly was to open each scheme and check. There was no reliable way to confirm nothing had been missed. An audit meant cross-referencing dozens of different configurations with no baseline to compare against.

Permission Model

One scheme covers all 55+ projects. Six role tiers define what each access level can do. Permission Groups, assigned at the platform level, map to those role tiers. When a user is onboarded, they're added to the group that matches their position. That group assignment flows through to every project automatically.

What I Built
01
Universal permission scheme
Designed and built a single scheme with six named role tiers: View Only, View and Comment, View Comment and Create, Member/Agent, Lead, and Manager/Admin. Each tier has a clear definition of what it allows. The names describe intent, not technical configuration, so project admins could understand the model without needing a reference guide.
02
Permission Group structure
Built a set of 60 to 70 Permission Groups mapped to role tiers in the scheme. Groups were named for positions and teams, not for individual projects. When someone was onboarded, they went into the group that matched their role. That single assignment gave them the right access across every project they needed to be in, without any project-level configuration required.
03
Full environment migration
Migrated all existing projects from their individual auto-generated schemes to the universal scheme. New projects created after the migration were cleaned up during setup — the auto-generated scheme was deleted and the universal scheme was applied before the project went live.
04
Onboarding integration
Onboarding requests came through the IT service desk. When a new user was provisioned, I added them to the correct Permission Group based on their department and position. If a position was new and the right group wasn't obvious, I contacted the manager directly. Access was consistent from day one without requiring follow-up configuration per project.
05
Quarterly access audits
Ran quarterly audits across all active users and projects. With one scheme, the audit was a check against a known model. Access either matched the scheme's definitions or it didn't. No cross-referencing dozens of different configurations to determine what was expected.

The Outcome

A permission change that used to mean editing 20+ individual schemes now means editing one. The change propagates to every project automatically. There's no risk of missing one, no need to audit whether the update made it through.

Project setup got faster. Applying permissions to a new project is a single step. The scheme exists, the groups are mapped, and the new project inherits the same model everything else uses.

Quarterly audits became meaningful instead of impractical. With one scheme defining what access is supposed to look like, an audit is a comparison. Before, there was no reliable baseline to compare against.

The model was also built to survive turnover. A new admin stepping in would find one scheme with clearly named roles, groups that match org positions, and a consistent structure across every project.
