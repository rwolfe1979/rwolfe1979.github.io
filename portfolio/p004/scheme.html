Overview

This page covers the design and administration of the universal permission scheme built to govern access across the entire Jira environment. The goal was to replace a fragmented per-project model with a single, consistent structure that was easy to maintain, audit, and hand off.

The final model: one permission scheme, six named role tiers, and 60 to 70 permission groups. Every project in the environment references the same scheme. Access is managed at the group level, not the individual level. A user's group assignment at onboarding determines their access across every project they belong to, without any project-level configuration required.

The Problem with Per-Project Schemes

Jira generates a permission scheme automatically when a project is created. By default, that scheme belongs only to that project. It works fine for a single project. At scale, it creates a management problem.

With 20+ auto-generated schemes in the environment, any change to how permissions worked had to be applied to each one manually. Atlassian platform updates occasionally introduced new permission types that needed explicit configuration. There was no reliable way to confirm that every scheme had been updated. An audit meant opening each scheme individually and comparing it against what the others had.

The risk wasn't just extra work. It was inconsistency. One project might be missing a permission that all the others had. One might have been configured slightly differently at creation and never corrected. With no single model to compare against, those gaps were easy to miss.

Scheme Design

The universal scheme was built around a principle: permissions should describe intent, not just grant access. Each role tier was named for what it's meant to be used for, so anyone looking at it (a project admin, a manager, a future Jira administrator) could understand the model without needing a technical explanation.

The scheme covers all Jira permission categories: issue access, project access, comment and attachment rights, workflow transitions, and project configuration. Role tiers are defined once in the scheme. Every project that references the scheme inherits those definitions automatically.

When Atlassian adds a new permission type in a platform update, the scheme is updated once. All 55+ projects reflect the change immediately.

Role Tier Definitions

Six tiers cover the full range of access needed across the organization. The tiers stack. Each higher tier includes the capabilities of the one below it plus additional rights.

Role Tier What They Can Do Typical Use
View Only Read access to the project. Can see issues and comments but cannot create, edit, or transition. Executives and cross-department stakeholders who need visibility without working in the project.
View and Comment Read access plus the ability to add comments to existing issues. Cannot create new issues. Reviewers and approvers who participate in discussions but don't submit or manage work.
View, Comment, and Create Read access, comments, and the ability to create new issues. Cannot edit or transition. Requestors and submitters, teams that use a service desk or intake project to submit work requests.
Member / Agent Full working access: create, edit, transition, and resolve issues. Standard for anyone doing active work in the project. Team members in Jira Software projects and agents in JSM service desks.
Lead Member-level access plus the ability to manage components, versions, and basic project settings. Cannot modify permission schemes or global settings. Team leads and department supervisors who manage their project's configuration without needing full admin rights.
Manager / Admin Full project administration rights within the project scope. Cannot modify global schemes or platform-level settings, those remain with the platform administrator. Department managers who own a project and need full control over how it's configured.

Permission Groups

Permission Groups are the mechanism that connects users to role tiers. Each group is named for the position or team it represents. A group is assigned to one of the six role tiers in the permission scheme. Any user added to that group automatically receives the access that tier defines, across every project where the group is applied.

The environment had between 60 and 70 Permission Groups. Groups weren't limited to job titles and departments. Some groups existed to populate user picker fields: dropdowns in forms and issue views that let people select other users. Supervisors, for example, were added to a dedicated supervisor group so their names appeared correctly in approver fields across the environment, even in projects they didn't actively work in.

Managing permissions at the group level rather than the individual level kept the model maintainable. When a team's access needs changed, the group assignment was updated once. All users in that group reflected the change. No individual user edits were needed.

Onboarding and Provisioning

New user requests came through the IT service desk. When IT processed a new hire, contractor, or returning employee, they submitted a Jira request with the user's name, email, department, position, and manager.

From there, the provisioning process was:

  1. Invite the user to Atlassian through Atlassian Access and correct their display name. Managed accounts in Atlassian Access sometimes pull in names incorrectly from the identity provider. This was a routine cleanup step.
  2. Grant the appropriate product access (Jira Software, JSM, Confluence, or a combination).
  3. Add the user to the Permission Group that matched their position.
  4. Confirm with the requestor once configuration was complete.

If the position was new to the organization and the correct group wasn't obvious, the manager was contacted directly. IT handled hardware and identity provisioning but didn't always have enough context about what a role required within Jira. Clarifying with the manager at onboarding was faster than correcting access after the fact.

Some users went into more than one group. Supervisors, for instance, were added to both their department group and a separate supervisor group. The department group handled their working access. The supervisor group made their name available in user picker fields across the environment.

Migration

When the universal scheme was built, the environment had more than 20 active projects, each with its own auto-generated scheme. All were migrated to the universal scheme. The migration was handled project by project, applying the new scheme and verifying that access behaved correctly before moving on.

New projects created after the migration were handled during initial setup. Jira's project creation process still generated an auto-created scheme. As part of setting up any new project, that scheme was deleted and the universal scheme was applied before the project went live.

Access Audits

Quarterly access audits reviewed all active user accounts and their Permission Group assignments. The audit checked:

  • Every active account matched a current employee or contractor
  • Group assignments were consistent with the user's current department and position
  • No user held Lead or Manager-level access without a current justification
  • Accounts inactive for 60 or more days were flagged for review

The single scheme made audits practical. Before the consolidation, confirming what access was supposed to look like required cross-referencing dozens of separate configurations. With one scheme, the model was known. The audit was a check against it, not a reconstruction of it.

If a project admin had assigned a user directly to a project role outside of the group structure, that assignment was reviewed during the audit and standardized if needed.

Project Admin Assignments

Project admins with Lead or Manager-level access could assign individual users directly to any role tier within their own project. That was intentional. Teams have legitimate needs to give someone temporary access, bring in a contractor, or adjust a specific user's rights without going through a formal request.

The platform administrator managed the Permission Group structure, the broader organizational model. Project admins managed individual assignments within their own scope. Both operated within the same permission scheme. The scheme's definitions didn't change based on how the assignment was made.

Individual project-level assignments were reviewed during quarterly audits. If an assignment appeared inconsistent with the user's role or no longer had a clear justification, the project lead was contacted for clarification.
